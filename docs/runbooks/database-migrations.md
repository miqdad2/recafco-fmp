# Database Migrations — RECAFCO FMP

This runbook covers the Prisma migration workflow for RECAFCO FMP. Read it before touching any migration command.

## Guiding rules

- **Never run `prisma migrate reset`** against `recafco_fmp_dev`, `recafco_fmp_app`, or any production database. This command drops the database. It is absent from all project scripts intentionally.
- **Never run `prisma db push`** against a shared or production database. It bypasses the migration history.
- **Never auto-migrate on API startup.** The API starts regardless of migration state; migrations are a deliberate manual step.
- **Migrations run only against named databases.** The dev migration workflow targets `recafco_fmp_dev`. Production deployments target the production database via `db:migrate:deploy`.
- **Test code uses `recafco_fmp_test` only.** Automated tests must never touch `recafco_fmp_dev` or production.
- Schema changes require a migration file tracked in version control. Ad-hoc DDL against the database is prohibited.

## Local development workflow

### Check current migration status

```bash
pnpm db:migrate:status
```

This lists applied and pending migrations for the database at `DATABASE_URL`. It requires a live database connection. Do not run this without authorized credentials.

### Create and apply a new migration

```bash
pnpm db:migrate:dev --name describe_the_change
```

- Creates a `.sql` migration file in `packages/database/prisma/migrations/`.
- Applies the migration to `recafco_fmp_dev`.
- Regenerates the Prisma client automatically.

**Before running:** confirm `DATABASE_URL` points to `recafco_fmp_dev` (never to production).

### Validate and format the schema

```bash
pnpm db:validate   # check schema.prisma for errors
pnpm db:format     # auto-format schema.prisma
```

### Regenerate the Prisma client (without migrating)

```bash
pnpm db:generate
```

Run this after pulling changes that include schema updates but no new migrations.

## Production deployment workflow

### Apply pending migrations to production

```bash
pnpm db:migrate:deploy
```

`migrate deploy` applies all pending migrations in order without creating new ones. It does not prompt and does not roll back automatically. This command is safe to run in a CI/CD pipeline against the production database.

**Steps before deploying:**
1. Confirm all new migrations are reviewed and committed.
2. Take a database backup.
3. Deploy the API build artifacts.
4. Run `pnpm db:migrate:deploy` with production `DATABASE_URL`.
5. Verify with `pnpm db:migrate:status`.
6. Confirm the API `/ready` endpoint returns 200.

## Prohibited commands

| Command | Reason |
|---|---|
| `prisma migrate reset` | Drops the database. Never run. |
| `prisma db push` | Bypasses migration history. Never run against shared databases. |
| Direct DDL (`ALTER TABLE`, `DROP TABLE`, etc.) | Breaks migration history. Use migrations instead. |
| `pnpm db:migrate:dev` against production | Creates unexpected migration files. Use `db:migrate:deploy` for production. |

## Schema location

```
packages/database/prisma/schema.prisma
packages/database/prisma/migrations/    ← committed migration files
```

## Migration file conventions

- Migration file names are auto-generated by Prisma from the `--name` flag.
- Each migration file contains only the SQL for that change — no DROP DATABASE, no TRUNCATE, no INSERT of seed data.
- Migration files must be committed to version control before they can be deployed.

## Owner / application role separation

The `recafco_fmp_app` role owns the database but should only have the permissions needed at runtime:

- `SELECT`, `INSERT`, `UPDATE`, `DELETE` on application tables
- `USAGE` on sequences

Schema DDL (CREATE TABLE, ALTER TABLE, etc.) should be performed by a migration-capable role, either the owner or a dedicated migration role. In simple local setups, `recafco_fmp_app` as owner handles this. For production, consult the DBA before changing role grants.

## Automated test migrations

Automated tests use `TEST_DATABASE_URL` pointing to `recafco_fmp_test`. Run migrations against the test database separately:

```bash
TEST_DATABASE_URL="postgresql://recafco_fmp_app:pass@localhost:5432/recafco_fmp_test" \
  pnpm --filter @recafco/database exec prisma migrate deploy
```

Test databases may be reset between test runs **only** if `recafco_fmp_test` is confirmed as the target.
