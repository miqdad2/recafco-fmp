# Architecture

## Style

Use a pnpm monorepo with a modular-monolith NestJS backend, a Next.js frontend, a background worker, and one PostgreSQL database with explicit module ownership.

Do not start with distributed microservices.

## Storage Model

### PostgreSQL

Business metadata, users, permissions, module records, comments, activity, approvals, notifications, audit logs, file metadata, SAP mappings, and sync status.

### MinIO

Documents, images, evidence, generated reports, and retained exports.

### Redis-Compatible Service

Queues, retries, locks, and short-lived cache only. It is never the sole source of truth.

### SAP Business One

Authoritative for SAP-owned business records.

Rules:

- No browser-to-SAP calls.
- No SAP credentials in frontend code.
- Read-only first.
- Test company before production.
- Direct HANA writes are prohibited.
- Future writes require idempotency and reconciliation.
- Integration failures must be visible to administrators.

## Deployment

Recommended internal hostname: `fmp.recafco.local`

Suggested PM2 names:

- `recafco-fmp-web`
- `recafco-fmp-api`
- `recafco-fmp-worker`

Use separate ports, database, Redis namespace, MinIO buckets, logs, and backups from all other applications.

## Database Architecture

### Package ownership

All Prisma and PostgreSQL access is owned by `packages/database`. No other package or application may instantiate `PrismaClient` directly.

### Connection model (Prisma 7)

Prisma 7 uses a driver adapter pattern. At runtime:
1. `pg.Pool` is constructed with connection URL, pool size, and timeout parameters from validated env.
2. `PrismaPg` adapter wraps the pool.
3. `PrismaClient({ adapter })` is constructed вЂ” no URL in schema.prisma.

For CLI operations (`prisma migrate`, `prisma generate`), the URL comes from `prisma.config.ts`.

### Role separation

- `recafco_fmp_app` вЂ” application runtime role; no superuser privileges; owns the application database.
- `postgres` superuser вЂ” used only during initial database/role setup; never used in application code or `.env`.

### Readiness model

`GET /ready` performs a bounded database health check on every call. The check uses a pg-level statement timeout (primary) plus a JS `Promise.race` timeout (secondary guard). If the database is unreachable, `/ready` returns 503 with `details: { database: 'unavailable' }`. Internal category (timeout, connection_refused, authentication, unknown) is logged only. `GET /health` always returns 200.

## Invariants

1. Permanent production software.
2. Existing systems must not be disrupted.
3. Never stop all Node.js processes.
4. Production data survives deployments.
5. Every protected operation is authorized on the backend.
6. Every material state change is auditable.
7. No direct SAP HANA writes.
8. SAP integration begins read-only.
9. Files live in MinIO; metadata lives in PostgreSQL.
10. Long-running work runs in the worker.
11. No mandatory paid SaaS without approval.
12. Missing requirements are documented, not guessed.
13. Never run `prisma migrate reset` against any company database.
14. Never use the `postgres` superuser as the runtime application account.
15. Migrations are deliberate manual steps; the API never auto-migrates on startup.
